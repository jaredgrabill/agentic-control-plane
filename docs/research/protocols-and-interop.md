## 1. Model Context Protocol (MCP)

**Spec status and versioning.** MCP uses date-stamped spec revisions. The current published revision is **2025-11-25** ([spec](https://modelcontextprotocol.io/specification/2025-11-25)). A **2026-07-28 revision is in release candidate** status — the largest revision since launch — with final publication scheduled for July 28, 2026, following a ten-week validation window in which Tier-1 SDKs must demonstrate support ([release-candidate post](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/), [2026 roadmap](https://blog.modelcontextprotocol.io/posts/2026-mcp-roadmap/)). Governance was formalized in the 2025-11-25 cycle (SEP process, Working Groups/Interest Groups, SDK tiering — SEP-932/1302/1730), and 2026-07-28 adds a formal feature lifecycle (Active → Deprecated → Removed, minimum 12 months between deprecation and removal). Notably, **Roots, Sampling, and Logging move to Deprecated** in the RC.

**Transports.** Two standard transports: **stdio** (local child-process servers) and **Streamable HTTP** (single endpoint, optional SSE streaming; it replaced the older HTTP+SSE dual-endpoint transport in 2025). The headline change in 2026-07-28 is a **stateless core**: the `initialize`/`initialized` handshake and `Mcp-Session-Id` header are removed; protocol version, client info, and capabilities travel in `_meta` on every request. Any request can hit any server replica — no sticky routing or shared session stores — making MCP servers deployable behind plain round-robin load balancers ([RC changelog](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/)). 2025-11-25 also allowed servers to close SSE streams at will with client polling/resumption via event IDs (SEP-1699).

**Auth.** Authorization is **OAuth 2.1-based** for HTTP transports: the MCP server is an OAuth resource server; discovery via RFC 9728 Protected Resource Metadata (`.well-known`, with optional `WWW-Authenticate`), plus OpenID Connect Discovery support and incremental scope consent (SEP-835), and **OAuth Client ID Metadata Documents** as the recommended alternative to Dynamic Client Registration (SEP-991) ([2025-11-25 changelog](https://modelcontextprotocol.io/specification/2025-11-25/changelog)). The 2026-07-28 RC hardens this with six proposals: mandatory `iss` validation (RFC 9207), `application_type` in DCR, and issuer-bound credential registration to prevent mix-up attacks in MCP's one-client/many-servers pattern.

**Other current-spec features worth knowing:** experimental **Tasks** for durable long-running requests with polling/deferred results (SEP-1686; becomes a first-class *extension* in 2026-07-28), tool calling inside sampling (SEP-1577), URL-mode elicitation (SEP-1036), icons metadata, JSON Schema 2020-12 as default dialect, and a first-class **extensions framework** in the RC with two official extensions: **MCP Apps** (sandboxed-iframe HTML UIs served by tools) and **Tasks**.

**Registry.** The **official MCP Registry** (registry.modelcontextprotocol.io) launched in preview September 8, 2025, is community-run, catalogs server metadata in a `server.json` format, and supports federation (sub-registries mirror/curate it). Still labeled preview (breaking changes/data resets possible), with ~9,600 latest server records and ~29,000 server/version records as of May 2026 ([GitHub](https://github.com/modelcontextprotocol/registry), [registry docs](https://registry.modelcontextprotocol.io/docs)).

**Enterprise deployment pattern.** The consolidated 2026 pattern is the **MCP gateway** — a control plane between agents and tool servers doing authN/Z, policy, credential brokering (token exchange for scoped downstream tokens), DLP, rate limiting, audit, and observability, solving the N×M client-server problem. Common topologies: centralized hub-and-spoke gateway cluster; federated meshes of team/region gateways under a central control plane; and "portal" designs with progressive tool disclosure (even collapsing all servers behind search/execute meta-tools) to cut token costs. Cloudflare published a reference architecture; a crowded vendor field (Tyk, TrueFoundry, Obot, MintMCP, etc.) exists ([Cloudflare](https://blog.cloudflare.com/enterprise-mcp/), [Tyk](https://tyk.io/learning-center/enterprise-mcp-gateway-key-considerations/)). Internally, enterprises run shared "tool server" platforms: a governed monorepo/template model with default-deny write controls, centralized secrets, and approval workflows. The stateless 2026-07-28 core materially simplifies this (no session affinity).

## 2. Agent-to-Agent (A2A) Protocol

**Status.** Launched by Google April 2025 (Apache-2.0), donated to the **Linux Foundation in June 2025**; TSC includes Google, Microsoft, AWS, Cisco, Salesforce, ServiceNow, SAP, and IBM. **v1.0.0 is the current stable spec** (prior: 0.3.0, 0.2.6, 0.1.0), released early 2026; as of April 2026 the LF reported 150+ production organizations and SDKs in Python, JavaScript, Java, Go, and .NET, with GA support in Microsoft Copilot Studio, Azure AI Foundry, and Amazon Bedrock AgentCore ([spec](https://a2a-protocol.org/latest/specification/), [1.0 announcement](https://a2a-protocol.org/latest/announcing-1.0/)).

**Agent Cards.** JSON capability manifests served at a well-known path (canonically `/.well-known/agent.json`; newer docs use `agent-card.json` — *exact filename in v1.0 unverified beyond secondary sources*). Cards declare name, version, description, service endpoints, supported protocol bindings, **skills** (typed capabilities with examples), input/output modes, and required **security schemes** (API key, HTTP Basic/Bearer, OAuth 2.0 authorization-code/client-credentials/device flows, OIDC, mTLS). v1.0's marquee addition is **Signed Agent Cards** — JWS signatures let a consumer verify a card was issued by the domain owner, the trust primitive for decentralized discovery. An "extended card" with richer detail can be gated behind authentication (*extended-card detail: training data, consistent with spec docs*).

**Transports and task lifecycle.** Three normative bindings: **JSON-RPC 2.0 over HTTP, gRPC (protobuf), and HTTP+JSON/REST**, plus custom bindings. Long-running work is modeled as **Tasks** with states: `submitted → working → {completed | failed | canceled | rejected}` (terminal), plus two interrupt states `input-required` and `auth-required` that pause without terminating. Real-time updates via **SSE streaming** and **push notifications** (webhooks to client-registered endpoints) for disconnected scenarios. Messages carry multi-part content (text, files, structured data) as Artifacts.

**Discovery model.** Deliberately registry-agnostic: well-known URI on the agent's domain, curated registries, or direct configuration. **There is no protocol-level central registry**; signed cards + declared auth schemes are the trust mechanism, and enterprises are expected to run their own catalogs ([Tyk guide](https://tyk.io/learning-center/a2a-protocol-architecture-and-technical-specification/)).

**A2A vs MCP.** Complementary layers, and both projects say so: **MCP is vertical** (agent ↔ tools/context — an agent's "hands"), **A2A is horizontal** (opaque peer agents negotiating multi-turn, long-running tasks without exposing internal state, memory, or tools). The canonical composition: an agent exposes an A2A Agent Card outward while consuming MCP servers inward; some platforms also wrap A2A agents *as* MCP tools for hosts that only speak MCP. Key semantic difference: MCP tool calls are (post-2026) stateless request/response with schemas known upfront; A2A tasks are stateful, negotiated, possibly hours-long interactions between autonomous parties.

## 3. Other Standards and Frameworks (brief)

- **ACP (IBM/BeeAI Agent Communication Protocol):** launched March 2025, **merged into A2A August 2025** under LF AI & Data — ACP is effectively end-of-life as a separate standard; BeeAI provides A2A adapters ([LF announcement](https://lfaidata.foundation/communityblog/2025/08/29/acp-joins-forces-with-a2a-under-the-linux-foundations-lf-ai-data/)). Treat any "ACP support" requirement as "A2A support."
- **AGNTCY:** Cisco-initiated (March 2025), moved to the **Linux Foundation July 29, 2025**, with Cisco, Dell, Google Cloud, Oracle, Red Hat as formative members. Components: **OASF (Open Agentic Schema Framework)** — an attribute-taxonomy schema for describing agent capabilities/metadata (think OCSF-for-agents); **Agent Directory Service (ADS)** — content-addressed, OCI-registry-based, cryptographically signed distributed agent discovery; **SLIM** messaging; identity; observability ([LF press release](https://www.linuxfoundation.org/press/linux-foundation-welcomes-the-agntcy-project-to-standardize-open-multi-agent-system-infrastructure-and-break-down-ai-agent-silos), [OASF docs](https://docs.agntcy.org/oasf/open-agentic-schema-framework/), [ADS paper](https://arxiv.org/pdf/2509.18787)). Adoption is far thinner than MCP/A2A; useful as a design reference for registries, not yet a must-implement.
- **ANP (Agent Network Protocol):** DID-based decentralized agent networking; academic/niche ([survey](https://arxiv.org/pdf/2505.02279)). *Adoption status: unverified/from training data.*
- **Microsoft Agent Framework:** AutoGen + Semantic Kernel converged into one SDK; public preview October 2025, **1.0 GA April 2026** (Python and .NET); AutoGen and SK are maintenance-mode. Graph-based workflows plus agent abstractions; native MCP and A2A support ([MS Learn](https://learn.microsoft.com/en-us/agent-framework/overview/)).
- **LangGraph:** 1.0 GA October 2025, now ~1.1.x; stateful-graph orchestration with checkpointing/human-in-the-loop; the de facto graph-orchestration pattern. **CrewAI:** role-based crews, now at 1.x with enterprise observability. **OpenAI Agents SDK:** handoff-based peer delegation, built-in MCP tool support, sessions/guardrails/tracing; major April 2026 overhaul (sandboxing, subagents); **no native A2A** ([docs](https://openai.github.io/openai-agents-python/)). **AgentKit** (DevDay, Oct 6 2025) layers a visual Agent Builder on top.
- **What an enterprise platform should adopt:** MCP for all tool access; A2A at agent boundaries; treat orchestration frameworks as swappable internals behind those two protocols. The stabilized framework taxonomy is graph-based (LangGraph/MS Agent Framework), role-based (CrewAI), handoff-based (OpenAI SDK), hierarchical (Google ADK) — large shops mix them, which is exactly why protocol boundaries matter more than framework choice.

## 4. Agent Registry & Discovery Patterns

State of the art is converging but **no single standard registry for agents exists yet** — A2A explicitly leaves it out of protocol scope. Patterns an enterprise control plane should implement:

- **Capability manifests:** A2A Agent Cards (skills + auth schemes + endpoints, JWS-signed in v1.0) are the interoperable format; OASF offers a richer taxonomy if you need structured capability classification. MCP's `server.json` plays the same role for tool servers. Internally, store a superset record: card + ownership, data classification, cost/model metadata, eval scores.
- **Semantic capability matching:** production systems combine structured filters (skill tags, I/O modes, auth compatibility) with embedding-based search over skill descriptions/examples; AGNTCY's ADS does content-addressed capability-centric lookup ([ADS paper](https://arxiv.org/pdf/2509.18787)). Pure semantic routing is unreliable alone — treat it as candidate retrieval, with deterministic policy filters as the gate. (*Assessment; specific failure rates unverified.*)
- **Dynamic registration/health:** the working pattern mirrors service registries (Consul/Kubernetes) but adds agent semantics: register on deploy with signed card; heartbeat/health endpoints checking not just liveness but *capability health* (canary prompts / synthetic task probes scoring output quality, since an agent can be "up" but degraded); deregister on TTL expiry. MCP registry supports versioned server records; Microsoft **Agent 365** (GA-track in 2026) is the flagship enterprise "agent registry as governed asset inventory," with cross-cloud registry sync (AWS Bedrock, Google Cloud) in public preview ([Futurum](https://futurumgroup.com/insights/microsoft-agent-365-turns-shadow-ai-into-a-governed-asset-class/)).
- **Versioning:** the emerging norm is **semver on the capability contract** (skill schemas, I/O formats) independent of the implementation version — model swaps or prompt changes that preserve the contract are patch/minor; schema-breaking changes are major, and registries keep multiple active versions with deprecation windows (mirroring MCP's new 12-month lifecycle policy). Tool/skill contract drift is widely cited as a leading cause of production agent failures (*specific "60%" figure from vendor blogs — unverified*).
- **Canary/shadow for agents specifically:** canarying agents differs from canarying services because outputs are non-deterministic — the deployment controller routes a traffic fraction to the new agent version for a soak window and gates promotion on *quality* metrics (LLM-judge scores, task success rate, escalation rate, cost/latency), not just error rates. **Shadow mode** runs the candidate agent on mirrored real inputs with side effects suppressed (writes stubbed, tool calls sandboxed) and diffs outputs against the incumbent offline. Both require the registry to hold multiple versions simultaneously with router-level weighting, plus session pinning so a multi-turn task doesn't switch versions mid-flight ([Sakura Sky lifecycle series](https://www.sakurasky.com/blog/missing-primitives-for-trustworthy-ai-part-11/); side-effect-suppression detail is standard practice but *unverified via primary spec*).

## 5. Agent Composition Patterns

- **Orchestrator–worker:** central agent decomposes a task, fans out to workers, synthesizes results. Best default for parallelizable research/analysis; Anthropic's multi-agent research system and most "subagent" primitives implement it. Key discipline: workers return conclusions, not raw context, to keep the orchestrator's window clean.
- **Planner–executor:** an explicit plan artifact is produced (often by a stronger model), then executed step-by-step (often by cheaper models), with replanning on failure. Use when tasks are long-horizon and auditability of intent matters; it makes human approval gates natural.
- **Blackboard:** agents share a common state store and opportunistically contribute; LangGraph's shared-state graph is the modern incarnation. Good for problems without a fixed decomposition; harder to govern — bound it with graph edges rather than fully open contribution.
- **Hierarchical delegation:** multi-level manager→sub-manager→specialist trees (CrewAI hierarchical process, Google ADK). Matches enterprise org-shaped problems and A2A naturally (each level is an opaque A2A peer), but each level adds latency, cost, and error compounding — keep hierarchies shallow.
- **Contract-net:** task announcement → bidding → award, from FIPA's classic Contract Net Interaction Protocol (SC00029 — *protocol lineage from training data*). Re-emerging for cross-organization A2A ecosystems where capability and price are negotiated dynamically. Overkill inside a single enterprise with a trusted registry.
- **Deterministic workflows vs LLM routing:** the settled guidance (Anthropic's "Building effective agents," Dec 2024 — *from training data*) still holds in 2026: **use deterministic, code-defined orchestration whenever the decomposition is known in advance** (predictable cost/latency, testable, auditable); reserve **LLM-driven routing/autonomous agents** for tasks where the step count or path is genuinely unknowable upfront. The pragmatic enterprise pattern is layered: deterministic workflow skeleton with LLM-routed decision points inside bounded nodes, semantic routing constrained to a registry-filtered candidate set, and hard policy/permission checks always enforced in code, never delegated to the model.
