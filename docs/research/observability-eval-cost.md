## 1. OpenTelemetry GenAI Semantic Conventions

**Status: active, widely adopted, but still not "stable."** As of mid-2026 the GenAI semantic conventions remain in *Development* (experimental) status with no published stabilization timeline, despite being the de facto interchange format for LLM telemetry ([OpenTelemetry blog, 2026](https://opentelemetry.io/blog/2026/genai-observability/); [Greptime state-of-play, May 2026](https://greptime.com/blogs/2026-05-09-opentelemetry-genai-semantic-conventions)). A significant structural change: the GenAI conventions have been **moved out of the main `semantic-conventions` repo into a dedicated repository, [`open-telemetry/semantic-conventions-genai`](https://github.com/open-telemetry/semantic-conventions-genai)**, covering spans, metrics, and events for GenAI clients, MCP (Model Context Protocol), and provider-specific conventions (OpenAI, Anthropic, AWS Bedrock, Azure AI Inference). That repo has 570+ commits but no tagged releases yet and an unfinished schema URL, confirming pre-1.0 maturity ([repo](https://github.com/open-telemetry/semantic-conventions-genai); [otel docs redirect page](https://opentelemetry.io/docs/specs/semconv/gen-ai/)).

**What is standardized (gen_ai.* namespace):**

- **Spans:** Each LLM call is a client span with `gen_ai.operation.name` (`chat`, `text_completion`, `generate_content`, plus `embeddings`), `gen_ai.request.model`, `gen_ai.response.finish_reasons`, and `gen_ai.usage.input_tokens` / `gen_ai.usage.output_tokens`. Agent conventions define lifecycle spans `create_agent` and `invoke_agent`, with child `chat` spans for LLM calls and `execute_tool` spans for tool invocations — giving a standard trace hierarchy for agent runs ([gen-ai spans spec](https://opentelemetry.io/docs/specs/semconv/gen-ai/gen-ai-spans/); [OTel blog](https://opentelemetry.io/blog/2026/genai-observability/)).
- **Metrics:** `gen_ai.client.operation.duration` (required latency histogram) and `gen_ai.client.token.usage` (recommended token histogram, dimensioned by `gen_ai.token.type` = input/output) ([OTel blog](https://opentelemetry.io/blog/2026/genai-observability/)).
- **Content capture:** prompt/completion bodies (`gen_ai.system_instructions`, `gen_ai.input.messages`, `gen_ai.output.messages`) are **opt-in and off by default** — only metadata (model, tokens, latency) is emitted unless explicitly enabled, which is the recommended privacy posture ([OTel blog](https://opentelemetry.io/blog/2026/genai-observability/)).
- **Migration mechanics:** semconv v1.36 is the transition baseline; instrumentations default to the older attribute format and switch to the latest via `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` ([Greptime](https://greptime.com/blogs/2026-05-09-opentelemetry-genai-semantic-conventions)).

**Enterprise implication:** build the platform's telemetry schema on `gen_ai.*` now (it's what every vendor ingests), but pin a semconv version and plan for breaking attribute renames until stabilization. OpenAI SDK instrumentation is the most mature; OpenLLMetry fills gaps for Anthropic, Cohere, Bedrock ([Greptime](https://greptime.com/blogs/2026-05-09-opentelemetry-genai-semantic-conventions)).

## 2. Tooling Landscape and Native Platform Capabilities

- **Langfuse** — leading open-source platform (self-hostable on Postgres + ClickHouse, OTel-native, framework-agnostic); strongest at operational telemetry, cost analytics, and prompt management. **Acquired by ClickHouse, Inc. in January 2026** (as part of ClickHouse's $400M Series D at ~$15B valuation); Langfuse remains open source and self-hostable with unchanged licensing, and claims 19 of the Fortune 50 as users ([ClickHouse announcement](https://clickhouse.com/blog/clickhouse-acquires-langfuse-open-source-llm-observability); [Langfuse blog](https://langfuse.com/blog/joining-clickhouse); [SiliconANGLE](https://siliconangle.com/2026/01/16/database-maker-clickhouse-raises-400m-acquires-ai-observability-startup-langfuse/)).
- **LangSmith** — fastest path to traces for LangChain/LangGraph shops; strongest agent-graph visualization but highest lock-in risk of the majors ([Digital Applied comparison](https://www.digitalapplied.com/blog/agent-observability-platforms-langsmith-langfuse-arize-2026)).
- **Arize Phoenix** — open-source, OTel-native (OpenInference), inherits classical-ML rigor: eval primitives, embeddings analysis, and drift detection that competitors approximate with weaker statistics; strongest for RAG evaluation ([Digital Applied](https://www.digitalapplied.com/blog/agent-observability-platforms-langsmith-langfuse-arize-2026); [MLflow comparison](https://mlflow.org/top-5-agent-observability-tools/)).
- **OpenLLMetry (Traceloop)** — Apache-2.0 OTel instrumentation set (Python/TS/Go/Ruby) covering LLM providers, vector DBs, and frameworks; exports to any OTel backend (Datadog, New Relic, Honeycomb, Langfuse), making it the neutral instrumentation layer rather than a destination ([GitHub](https://github.com/traceloop/openllmetry); [Traceloop docs](https://www.traceloop.com/docs/openllmetry/introduction)).
- **W&B Weave** — best fit where W&B already owns experiment tracking; strongest for experimentation/prompt-research workflows, weaker as a standalone production observability choice ([Kanerika comparison](https://kanerika.com/blogs/llmops-observability/)).

**Capabilities an enterprise platform should provide natively** (the common denominator across these tools): hierarchical trace capture of full agent runs (LLM calls, tool calls, retrievals) on OTel/`gen_ai.*`; session/conversation grouping with replay of multi-turn interactions; token- and dollar-accurate cost accounting per trace computed at span close; user/human feedback capture (thumbs, scores, annotations) attached to traces; dataset creation from production traces; prompt/config version tracking; and evaluator execution against both datasets and live traffic ([Latitude tool comparison](https://latitude.so/blog/best-ai-agent-observability-tools-2026-comparison); [Braintrust cost-tracking playbook](https://www.braintrust.dev/articles/how-to-track-llm-costs-2026)). Pricing at ~1M traces/month, self-host option, and data residency are the usual enterprise selection axes ([Digital Applied](https://www.digitalapplied.com/blog/agent-observability-platforms-langsmith-langfuse-arize-2026)).

## 3. Evaluation Practices

**Offline (golden datasets + CI gates).** Consensus practice is a "four-bucket" golden set — production samples, adversarial cases, edge cases, and replayed failures — so a CI gate actually predicts production behavior ([qaskills golden-dataset guide](https://qaskills.sh/blog/golden-dataset-llm-evaluation-guide)). On every PR touching prompts/models/pipelines, run the eval suite, compare to a stored baseline, and block merge on regression beyond tolerance; the LLM judge used should be calibrated to ~85–90% agreement with human annotations before it is trusted as a gate ([FutureAGI regression testing](https://futureagi.com/glossary/llm-regression-testing/); [DevToolLab evals guide](https://devtoollab.com/blog/llm-evals-guide-2026)). Common tooling: promptfoo (YAML tests, CI diff reports), DeepEval/Confident AI (hosted regression suites), plus native evaluators in Langfuse/LangSmith/Phoenix ([DeepEval LLM-as-judge guide](https://deepeval.com/blog/llm-as-a-judge)).

**Online (LLM-as-judge on sampled traffic).** Score ~5–10% of production traffic with automated evaluators (judges, guardrail classifiers) and watch trends; scoring 100% is cost-prohibitive. The key pattern is a **shared rubric**: the same judge template runs offline against the golden set and online against sampled traces, making offline and online scores comparable ([qaskills offline-vs-online](https://qaskills.sh/blog/offline-vs-online-llm-evaluation-2026); [TrueFoundry gateway-based online eval](https://www.truefoundry.com/blog/online-llm-evaluation-gateway)). Online failures feed back into the golden set (failure replays), closing the loop.

**Drift detection for agents.** Beyond input/data drift, teams now monitor *behavioral drift* (changes in tool selection, decision patterns) and *semantic drift* in outputs via embeddings of queries/responses. 2026 best practice is to log embeddings + evaluator scores on every production trace and **alert on the joint condition of input drift plus eval-score drop**, which sharply reduces alert noise versus alerting on either alone ([Galileo drift monitoring](https://galileo.ai/blog/best-llm-output-drift-monitoring-platforms); [ElixirData agent drift](https://www.elixirdata.co/blog/ai-agent-drift-detection)). A January 2026 "Agent Stability Index" (12-dimension drift framework) and arXiv work on black-box persona drift detection exist but are early-stage ([Syrin](https://www.syrin.ai/blog/agent-drift-quantifiable); [arXiv 2605.09863](https://arxiv.org/pdf/2605.09863)).

**Eval-gated deployment.** The emerging standard pipeline is five gates: lint → offline eval → cost budget check → shadow eval on production traces → canary with auto-rollback ([AppScale AI-native CI/CD](https://appscale.blog/en/blog/ai-native-cicd-for-llm-features-eval-gates-prompt-diff-canary-rollouts-2026)). For agent versions specifically, a four-stage rollout is common: **shadow → canary → percentage ramp → full** ([FutureAGI agent rollout](https://futureagi.com/blog/agent-rollout-strategies-2026/)).

**Shadow-mode comparison.** Implemented at the gateway via request duplication: production version serves the user while the candidate receives a forked copy of the same request; both responses carry a shared correlation ID and are scored on identical judge templates, enabling paired comparison with zero user risk ([FutureAGI shadow/canary](https://futureagi.com/blog/llm-eval-shadow-traffic-canary-2026/)).

**A/B and canary analysis.** Canaries reuse the CI judge templates on live traffic so regressions that slipped past offline evals surface before the next ramp step; auto-rollback triggers on score/latency/cost breaches. A/B testing of prompt/model variants against feedback and judge scores is a native feature in Confident AI, LangSmith, and similar platforms ([AppScale](https://appscale.blog/en/blog/ai-native-cicd-for-llm-features-eval-gates-prompt-diff-canary-rollouts-2026); [Confident AI](https://www.confident-ai.com/knowledge-base/compare/top-langsmith-alternatives-and-competitors-compared)).

**Red-teaming automation.** Automated adversarial testing is now a **blocking CI gate**: each change triggers scans for prompt injection, PII leakage, jailbreaks, and system-prompt extraction. Tools: promptfoo red-team mode, DeepTeam/Confident AI, Galileo, Mindgard (recon → automated adversarial testing → runtime guardrails) ([Galileo red-teaming strategies](https://galileo.ai/blog/llm-red-teaming-strategies); [Confident AI red-teaming tools](https://www.confident-ai.com/knowledge-base/compare/best-ai-red-teaming-tools-2026)). Microsoft PyRIT is also standard here (from training data — not re-verified in this pass).

## 4. Cost Optimization / FinOps for GenAI

**FinOps Foundation guidance.** The Foundation has made AI its top priority, updating its mission to "the Value of Technology"; the State of FinOps 2026 reports **98% of practitioners now manage AI spend, up from 31% two years prior**, and AI cost management is the #1 skill gap ([data.finops.org](https://data.finops.org/); [FinOps for AI overview, updated Feb 2026](https://www.finops.org/wg/finops-for-ai-overview/)). Its FinOps-for-AI framework treats AI as a distinct cost domain with: unit economics (cost per inference, cost per token, ROI vs. business value), a crawl-walk-run maturity path (budget-capped pilots → basic tracking/anomaly detection → deep budget segmentation with SLA-aware optimization), **showback before chargeback**, and tagging by project/environment/team/cost-center. It flags the hardest problem as attributing shared-model output to consuming features ([FinOps for AI overview](https://www.finops.org/wg/finops-for-ai-overview/); [framework category page](https://www.finops.org/framework/technology-categories/ai/)).

**Optimization levers (typical published magnitudes):**

- **Prompt caching** — highest-leverage, lowest-risk lever: up to ~90% off input tokens on repeated prefixes; an academic study across 500+ agent sessions with 10K-token system prompts measured 41–80% cost reduction and 13–31% faster time-to-first-token. Requires cache-aware prompt layout (static prefix first, volatile content last) ([Digital Applied prompt caching](https://www.digitalapplied.com/blog/prompt-caching-2026-cut-llm-costs-engineering-guide); [DevToolLab](https://devtoollab.com/blog/prompt-caching-guide)).
- **Model routing / cascades** — classify difficulty and send easy requests to small models (small-model-first with escalation-on-failure); typical 40–70% savings ([Mavik Labs](https://www.maviklabs.com/blog/llm-cost-optimization-2026); [Morph](https://www.morphllm.com/llm-cost-optimization)).
- **Batch APIs** — Anthropic and OpenAI both offer ~50% discounts for async processing (results within 24h); "free" savings for any non-interactive workload ([Mavik Labs](https://www.maviklabs.com/blog/llm-cost-optimization-2026)).
- **Semantic caching** — embed the query, serve cached responses above a similarity threshold; effective for FAQ-like traffic, with well-known false-hit risks requiring threshold tuning ([GMI Cloud](https://www.gmicloud.ai/en/blog/llm-inference-cost-optimization-caching-batching-routing)).
- Combined, vendors claim 70–85% total spend reduction; treat specific percentages as directional vendor figures.

**Token budgeting, attribution, and quotas.** Per-tenant/per-feature attribution is described as "the prerequisite for every other cost lever" — routing, caching, and rate limits can't be targeted until you know who spent what ([Particula per-tenant attribution](https://particula.tech/blog/per-tenant-llm-cost-attribution-multi-tenant-saas)). Standard architecture: metadata tags (user, tenant, feature, agent, environment) injected at the gateway; cost computed at span close from `gen_ai.usage.*`; three-tier rollups (real-time → hourly → daily chargeback source-of-truth); CSV export into corporate FinOps tooling ([TrueFoundry cost attribution](https://www.truefoundry.com/blog/llm-cost-attribution-team-budgets); [Braintrust](https://www.braintrust.dev/articles/how-to-track-llm-costs-2026)). Enforcement patterns: **hierarchical budgets** (virtual key → team → customer → org) with auto-reset windows; soft budgets (alerts) vs. hard caps returning structured 429s the client degrades gracefully on; token- and request-rate limits per virtual key; and policy-driven automatic downgrade of cost-sensitive consumers to cheaper models. Gateways (LiteLLM, TrueFoundry, agentgateway — which shipped LLM consumption/chargeback features in v1.3.0, June 2026) are the natural enforcement point since attribution, auth, and rate limiting share one policy engine ([agentgateway v1.3.0](https://agentgateway.dev/blog/2026-06-17-agentgateway-v1.3.0/); [Maxim gateway roundup](https://www.getmaxim.ai/articles/5-enterprise-ai-gateways-for-llm-cost-control-in-2026/)).

## 5. SLOs for Agent Systems

Agent reliability is multi-dimensional: beyond availability and latency, SLOs must cover **task success rate, tool-call reliability, output quality, escalation behavior, and cost per task** — each with its own SLI and its own parallel error budget (`error_budget = 1 − slo_target`) ([Zylos SRE for agents](https://zylos.ai/research/2026-03-22-sre-ai-agent-systems-observability-incident-response/); [Fast.io agent SLO guide](https://fast.io/resources/ai-agent-slo-management/)). Notable adaptations of classical SRE:

- **Error budgets on accuracy/quality:** define a quality SLI from judge scores or human-verified task success on a sampled basis; burning the quality budget freezes prompt/model changes exactly as an availability budget freezes deploys. An emerging "error budgets 2.0" pattern has the agent runtime monitor its own remaining budget and throttle or escalate autonomously ([BuildMVPFast error budgets](https://www.buildmvpfast.com/blog/ai-agent-error-budget-sre-reliability-autonomous-2026)).
- **Behavioral correctness SLIs:** for each agent decision class, set a human-review sample rate and track the agent-vs-human-override delta as an SLO input; accuracy and *consistency* are distinct metrics, and predictability deserves its own SLO ([DEV: SLO design for agentic systems](https://dev.to/ajaydevineni/slo-design-for-agentic-ai-systems-why-traditional-reliability-metrics-break-and-what-to-use-3581); [Microsoft: SRE for autonomous agents](https://techcommunity.microsoft.com/blog/linuxandopensourceblog/applying-site-reliability-engineering-to-autonomous-ai-agents/4521357)).
- **Cost SLOs:** cost-per-completed-task (total spend ÷ completed tasks) as a budget SLI, with alerting before invoice time ([Fast.io](https://fast.io/resources/ai-agent-slo-management/)).
- **Latency:** percentile SLOs on time-to-first-token and end-to-end task completion, per agent operation (from the `gen_ai.client.operation.duration` histogram).
- **Alerting on behavioral regressions:** alert on joint drift + eval-score-drop conditions (Section 3), rising thumbs-down/guardrail-flag rates, and tool-error spikes; and track *recoverability* metrics — mean time to detect unsafe output, mean time to disable an agent, rollback time, human-override rate ([Zylos](https://zylos.ai/research/2026-03-22-sre-ai-agent-systems-observability-incident-response/)).

## Caveats

- Specific savings percentages (40–70% routing, 70–85% combined) are vendor/blog figures, directional not audited.
- OTel GenAI attribute names verified against the official blog and spec pages; exact per-attribute stability levels change frequently — check [`semantic-conventions-genai`](https://github.com/open-telemetry/semantic-conventions-genai) before hard-coding.
- PyRIT as a red-teaming standard, and current Anthropic/OpenAI cache-pricing specifics (e.g., cache-write surcharges, TTL tiers), are **from training data** and were not re-verified in this pass.
- Several SLO/eval sources are practitioner blogs rather than peer-reviewed or foundation-issued guidance; the FinOps Foundation and OpenTelemetry citations are the authoritative anchors here.
